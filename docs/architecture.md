# Architecture

`jeryu-tool-finder` is the **tool-discovery** arm of the jeryu split family. It
owns no product code and ships no binary; it is pure Python scripts plus docs
that drive a runtime engine and file proposals into the `jeryu-tool` registry.

## Boundaries

This repo is a **public-portal / scripts-only** profile (see
`agent/boundaries.toml`): all Python lives under `scripts/` (the discovery
surface) and `ops/` (the CI gate), declared as `allowed_non_product_paths`. It
takes **no Cargo dependency** — the cross-repo clustering engine lives in
`jeryu-intelligence/crates/jeryu-codegraph` and is invoked as a binary at
runtime, so this repo adds no version-pin edge to the family graph.

## Components

- **`scripts/scan_family.py`** — resolves the codegraph engine
  (`JERYU_CODEGRAPH_BIN` → `PATH` → a prebuilt `jeryu-intelligence/target/*` →
  `cargo run`) and runs `tool-build scan-family --min-repos 2` over
  `repos.manifest.toml`, writing `dossiers/clusters.json`.
- **`scripts/dossier.py`** — enriches each cross-repo cluster into a dossier
  (per-repo file paths and line ranges, normalized preview, suggested tool
  kind/name, anticipated LOC saved). Carries a hermetic `--selftest` over
  `fixtures/sample-clusters.json` so the enrichment logic is provable offline.
- **`scripts/propose.py`** — promotes a chosen cluster into a `jeryu-tool`
  proposal (`[[tool]]` with `status=proposed` plus a `tasks/NNNN-*.toml` build
  task). Idempotent on `origin_cluster`. This is the one script that mutates a
  sibling repo.
- **`scripts/registry_summary.py`** — reads the golden-box registry summary back
  from `jeryu-tool` (CLI/MCP convenience).
- **`ops/ci/`** — the deterministic CI gate (`check` / `score` / `security`) and
  the shared `lib.sh` (tool + pinned-auditor guards).

## Data flow

```
repos.manifest.toml
   → scan_family.py  → dossiers/clusters.json   (cross-repo clusters, repo_count >= 2)
   → dossier.py      → dossiers/<cluster>.md + index.json
   → (agent/human decision)
   → propose.py      → jeryu-tool: [[tool]] (status=proposed) + tasks/NNNN-*.toml
```

`dossiers/` is a generated zone (`agent/generated-zones.toml`), regenerated by
`just scan && just dossier`; it is gitignored and never hand-edited. The dossier
schema and the LOC-saved definition live in [`tool-finder.md`](tool-finder.md).

## Ownership & proof routing

Ownership of every path is declared in `agent/owner-map.json`; the proof route
for every source path (which lane re-verifies it) is declared in
`agent/test-map.json` and the lanes themselves in `agent/proof-lanes.toml`. See
[`testing.md`](testing.md) for how the lanes run locally and in CI.
